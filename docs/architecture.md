# CursorialEdit Core Architecture — Final Specification

**Status: APPROVED by the maintainer (Mike), 2026-07-03.** Binding for implementation; §6 alternatives
are settled — do not relitigate without new evidence.

**Provenance.** All three judges (perf, fidelity, delivery) ranked **hybrid** first; this specification is hybrid with eight grafts the judges named from **engine** and **leverage**, and explicit resolutions for every objection raised against hybrid. Objections against the two losing proposals are recorded in §6 (Rejected alternatives) so they are not relitigated.

---

## 1. Decision summary — binding architectural decisions

1. **The source text is the only truth.** A line table (`List<Line>` where `Line = { string Text; LineEnding Ending }`) is the canonical artifact; AST, presenters, run maps, and table models are derived, invalidatable overlays — saves write the buffer, never a serialization of the AST, so §2 round-trip holds by construction.
2. **Markdig 1.3.2 is the parser**, pipeline pinned (`UsePipeTables, UseTaskLists, UseEmphasisExtras(Strikethrough), UseAutoLinks, UseFootnotes, UseDefinitionLists, UseAlertBlocks, UseMathematics, UseYamlFrontMatter, UsePreciseSourceLocation`); a custom parser reinvents a solved problem and concentrates the project's §2 correctness risk in unwritten code.
3. **Incrementality is an app-owned reparse window around Markdig**, with a conservative always-on window path and a fuzz-verified fast path gated on *both inserted and removed* text; full reparse only on load and large structural paste (§13 [DECISION] sanctions this).
4. **Block identity survives edits via engine's re-adoption rule** — match by (kind, first unmodified line) using per-line `Version` stamps, *not* content hashes, because the edited block's hash changes on every keystroke (unanimous judge objection, resolved).
5. **Inline parsing is lazy** (graft from engine): edits re-segment blocks eagerly, but inline runs are computed only when a block is realized into the render band or queried (find/outline) — bounding open/paste cost to segmentation plus the visible band.
6. **The surface is a custom `EditorControl : Control` templating its own `ScrollViewer`** (mandatory: the `HandlesScrolling` gate checks `TemplatedParent` — `Cursorial.UI/Controls/Control.cs:118-121`) whose SCP direct content is a `DocumentPanel : Panel, IScrollContentHost` (`Cursorial.UI/Controls/IScrollContentHost.cs:15`, public seam), `IsLogicalScroll = false` (the legacy step is already one cell / one viewport — `IScrollContentHost.cs:65-69`).
7. **One presenter element per *leaf* block, each `IsRenderBoundary = true`** (`Cursorial.UI/UIElement.Properties.cs:116-117,196-197`; promotion sticky until detach, `RenderTree.cs:37`): lists and blockquotes decompose to per-item/per-line presenters, tables get per-row boundaries in M3, large code fences per-line-group boundaries — so a keystroke re-rasters one small zone, never a band.
8. **The caret is a source anchor rendered through per-visual-row run maps with block-relative spans**; hidden marks are zero-width runs at true source positions, synthetic glyphs map to marker source or nothing — unedited blocks' maps stay valid across edits with zero rewriting.
9. **Reveal-on-edit — per-kind policy** (revised 2026-07-05; originally "never re-wraps"). Two reveal modes: **slide** — the active line renders at natural width, horizontally slid inside the block's box with grapheme-snapped clipping, line count invariant so no other line moves (§4.1 [EDGE] satisfied structurally); **wrap-reveal** — the active line wraps in place under the block's wrap mode with its marks shown, and the active block *reflows* (its height varies while active, content below shifts via the panel's prefix sums), so a prose paragraph keeps its surrounding context while edited. **Code blocks and raw mode always slide** (reflowing structured text is wrong); **prose (paragraph/heading/quote/list) wrap-reveals** when the "wrap while editing" option is on (default) and the block wraps — else it slides. **Table cells reveal per-CELL (M3, retiring deferred-cleanup #4):** an *inactive* cell renders its inline constructs FORMATTED — `` `code` ``/`**bold**`/`*italic*`/`~~strike~~`/`[text](url)` show their marks HIDDEN and their content styled exactly as the prose paragraph path styles the same inline kinds — while the one cell the caret is in reveals its RAW markdown so the user edits in place. The reveal granularity is the CELL: because hiding marks makes an inactive cell *narrower*, the active cell (raw, wider) may wrap to more visual rows, so its row *reflows* when the caret enters/leaves it (the grid column width, measured over the marks-hidden DISPLAY widths, is unchanged — only the active row's height varies, so no sibling column moves). Editing, copy, and every structural op stay on the RAW source (cell spans/`CellContent`/`CellContentRange` are unchanged); only the inactive rendering + the per-cell display↔source caret map are added. A plain cell (no inline marks — the common case) renders byte-identically to before, taking the raw path with no projection. The table's *horizontal* overflow (column-window, §2.5) is orthogonal and still slides the whole grid. Only the *active* block's height varies (blocks above are unmoved); the caret stays visible by horizontal slide (slide mode) or ordinary vertical scroll-follow (wrap mode). The `LeafBlockPresenter` primitive defaults to slide (the reviewed M2 reveal tests hold); the production `MarkdownViewBridge` opts prose in via `RevealWraps`/`EditWrapEnabled`. Two View commands (M5): "wrap for display" (master) and "wrap while editing" (nested under it) — see docs/framework-feedback.md FB-27 for the `CheckableCommandParameter` mechanism the latter uses.
10. **Undo is one document-scoped stack of coalescing edit groups** (750 ms idle seal via frame-aligned `UITimer`, `Cursorial.UI/Animation/UITimer.cs:18`; caret state restored per §3.3 [EDGE]); `TextBox`'s splice+coalesce records (`Controls/TextBox.cs:114-118, 683-713`) are the reference shape, reimplemented at document scope.
11. **Tables are `TablePresenter`s over a `TableModel`** with cell-measured column widths (`GraphemeWidth.ClusterWidth/StringWidth`, `Cursorial.Core/Text/GraphemeWidth.cs:31,106,172`) that are **viewport-aware, not a hard `[3,40]` clamp** (revised 2026-07-05). The model exposes each column's *natural* width (`MaxContentWidth` floored at `MinWidth`, no cap) plus the `(MaxContentWidth, CountAtMax)` cache; `TablePresenter` — at **measure** time, where the available viewport width is finally known (`MeasureOverride`'s `availableSize`), never at parse — runs browser-like auto-layout (`TableModel.ResolveColumnWidths` over the viewport budget minus grid chrome): a table that **fits** grows its columns to content (a column may exceed the old 40 cap, no wrap); one that **overflows** shrinks the widest column(s) toward a common ceiling and **word-wraps** their cells, never below `MinWidth`. A viewport **resize** re-resolves the widths and re-lays-out the table through the same per-row reconcile as a content-driven width change (a geometry change bounded to the table, never the document). The old `MaxWidth = 40` survives only as a viewport-unaware fallback (pre-measure / off-band caret map). Cell content **word-wraps** (`CaretNavigator.Wrap` / `WrapMode.WordWrap`, the same wrap prose uses — the class now lives in the Document project so `TableModel.WrapCell` can reuse it) with a char-level fallback only for a single over-long word; `CountAtMax` stays the O(1) shrink hint, and **column-window horizontal overflow is rendered inside the presenter** (WP6) — no nested ScrollViewer, sidestepping FB-6 entirely.
12. **Raw mode is a presenter mode, split view is two views over one model**: same `Document`, `BlockList`, undo stack, and edit pipeline; run maps degenerate to identity in raw mode, so caret preservation across toggles is the ordinary mapping (§4.2 [EDGE]s fall out).
13. **All derived data is epoch-stamped** (graft from engine): off-thread parse/layout results carry the `Document.Epoch` they were computed at and are rejected on mismatch — stale-span bugs fail loudly, never corrupt silently.
14. **The Markdig-oracle differential harness is a first-class deliverable**: it fuzzes windowed-vs-full parses, asserts *span fidelity* (each precise inline span's source slice must reproduce its construct — closing the shared blind spot the fidelity judge named), and generates the §2 acceptance conformance document.

## 2. Design narrative

### 2.1 Document model

`DocumentBuffer` holds the line table. Rationale over rope/gap-buffer: parsing, reveal (§4.1), find, and the raw view are all line-oriented; at the §10.2 ceiling (~10k lines / 1 MB) a `List<Line>` splice is a sub-millisecond reference memmove, and per-line `LineEnding` preserves mixed LF/CRLF byte-exactly (§10.1). `IDocumentBuffer` is a seam; a rope is a swap-in if the size target ever grows.

The canonical coordinate is `TextPosition(int Line, int Col)` (UTF-16 column; cell math lives in layout). Absolute offsets are derived via a prefix-sum cache invalidated from the edit line. **Offset stability is two-level, not marker-tree:**

- Blocks store `LineCount`, not `StartLine`; start lines and rendered heights are prefix sums over the `BlockList`, recomputed from the edit point (O(blocks), trivial at ~2k blocks). An edit inside block *i* bumps only block *i*; everything after shifts implicitly.
- Inline spans are **block-relative** against the block's source snapshot: an unedited block's spans never need rewriting; an edited block is reparsed anyway. (This is why leverage's absolute-offset run maps were rejected — see §6.)

A small `AnchorTable` (position + gravity, adjusted per edit) covers long-lived non-block positions: find-match highlights, footnote back-references, autosave dirty ranges. Each line carries a `Version` stamp bumped on mutation, and the document carries an `Epoch` bumped per applied edit group — the inputs to block re-adoption (§2.2) and the staleness guard (Decision 13).

**Edits** funnel through one primitive: `EditController.Apply(Edit(TextPosition Start, string Removed, string Inserted), EditKind)` — keystrokes, commands, paste, table ops, replace-all, and undo/redo replay (flagged non-recording) all take the same path, so anchors, parser, and every view update through one pipeline. **Undo groups** coalesce adjacent typing; seal on caret motion not caused by the edit, newline, any structural command (each its own unit per §3.3), or 750 ms idle (`UITimer`, deterministic under the fake clock in tests). Depth N=1000 configurable, oldest discarded; redo clears on new edit; groups restore `CaretBefore` including selection.

**Clipboard:** copy/cut serialize the selection's source range (§3.4) via `IClipboardService` (OSC 52 write). Paste arrives as bracketed paste → `TextInput { FromPaste = true }` (FB-3: reads are blind); an app-internal clipboard store makes in-app Copy→Paste round-trip regardless. Plain paste is a mode/command rather than `Ctrl+Shift+V` alone (legacy wire drops Shift — FB-14), with the chord bound additionally on Kitty-capable wires.

### 2.2 Parser and incrementality

**Coverage** (verified against Markdig at the pinned version): pipe tables, task lists, strikethrough, autolinks matching §2.2's scheme list, footnotes, definition lists, GitHub alerts (§2.3 callouts), math with the no-space `$` rule (§2.3 [EDGE]), YAML front matter, and spec-conformant CommonMark core including setext disambiguation (§2.1 [EDGE]) — the largest correctness surface in the project, retired by dependency. Markdig's *round-trip* machinery (trivia tracking) is deliberately unused: we never serialize from the AST. §2.1 [DECISION] normalizations (setext→ATX, indented→fenced, bullet→`-`) are targeted line rewrites at save time driven by block spans (`SaveNormalizer`), each unit-testable. **Resolution of the fidelity judge's table-padding objection:** inter-pipe padding is normalized **only for rows the user actually edited or tables modified by structural commands** — untouched tables round-trip byte-for-byte, consistent with the conservative reading of §17 Q4 pending the maintainer's answer.

**Incremental strategy (§13):**

1. **Window path (default).** Seed = blocks overlapping the edited lines, plus the previous top-level block (setext underlines, lazy continuation) plus the next (list/quote joins); expand outward while inside any container. Containment queries use a **document-level interval set of open fence / front-matter / `$$` regions** (graft from engine) for O(log n) safe-start lookup instead of prose-rule scanning. Parse the window text through the full pipeline off the fast path's budget — a few KB, well under 1 ms.
2. **Fast path (optimization only).** Skipping window expansion is permitted only when **both the inserted and the removed text** are free of boundary-significant content — newlines, fences, `|`, `#`, `>`, list markers, `---`/`===`, `$$`, `:` at line start (definition lists), `[^` (footnote definitions), and reference-definition-shaped lines. This resolves the perf judge's deletion hole (backspacing a blank line merges paragraphs while "introducing" nothing) and the delivery judge's blacklist gaps (`:`, `[^`, link-ref definitions): the gate inspects removed text symmetrically, the character set is a conservative superset, and the differential fuzzer (below) exercises the fast path explicitly. If the fuzzer ever shows the gate losing, the fallback is simply "always window" — measured, the window path alone meets §13.
3. **Fence parity.** If the window's trailing state flips between inside/outside an open fence, front matter, or `$$`, extend to EOF — the rare structural keystroke legitimately reinterprets the tail; segmentation is cheap line scanning and inline parsing stays lazy (Decision 5).
4. **Document-global constructs.** A `LinkRefTable`/`FootnoteTable` with a label→referencing-blocks index invalidates exactly the referencing blocks on definition change. When a window edit touches the definition set, the frame is served from the windowed result and a **debounced full reparse runs off-thread** (graft from leverage), posting back via `UIDispatcher.InvokeAsync` with epoch validation.
5. **Block identity — the re-adoption rule (graft from engine, replacing content hashes).** Outside the window, blocks keep their `BlockId` with implicit line-shift (Decision 8's prefix sums). Inside it, new segments match old blocks by **(kind, first unmodified line)** — a line is unmodified when its `Version` ≤ the block's recorded `ContentStamp` — with common prefix/suffix trimming; matched blocks are re-adopted (same `BlockId`, presenter, layout cache, fold state, table cell focus), so the block being typed in survives every keystroke. This resolves the objection all three judges raised against hash-based identity.

The parser emits `BlockListChange { Reused, Changed, Added, Removed, LineShift }`; the view reconciles presenters by `BlockId`, re-measures only changed ones, updates height prefix sums, and calls the public `ScrollContentPresenter.InvalidateScrollExtent()` (`Controls/ScrollContentPresenter.cs:330`) when the extent moves.

**The differential harness (Decision 14)** is built in M2 week 1, before the renderer: (a) random documents × random edit scripts, asserting windowed-splice ≡ full-parse (kinds, spans, line ranges) with fast-path edits explicitly in the mix; (b) **span-vs-source oracle** — every `UsePreciseSourceLocation` inline span's text slice must reproduce its construct, across all enabled extensions, on the CommonMark/GFM corpora — because windowed-vs-full comparison alone cannot detect spans that are *consistently* wrong (fidelity judge's shared objection, accepted and closed); (c) the passing-sections report becomes the §2 acceptance conformance document.

### 2.3 Rendering/editing surface

`EditorControl : Control` overrides `protected override bool HandlesScrolling /* declared protected internal; cross-assembly overrides must narrow to protected (CS0507) */ => true` and templates its own `ScrollViewer` (the gate checks `TemplatedParent` — `Control.cs:118-121`), whose SCP direct content is `DocumentPanel : Panel, IScrollContentHost`: `IsScrollClient = true`; `ScrollOwner` SCP-injected (`IScrollContentHost.cs:24-28`); `GetExtent()` from the block-height prefix sums (realized exact, unrealized estimated by line count × wrap factor, refined via `InvalidateScrollExtent`); `SetViewport` recorded; `InvalidateRealization()` marks the band dirty; `IsLogicalScroll = false` — verified against the seam's own docs that the legacy path already steps one cell / one viewport (`IScrollContentHost.cs:65-69`), resolving the confusion the judges caught in engine. The offset stays SCP-owned (the framework's deliberate `IScrollInfo` deviation).

**Band management:** realize viewport + 2K, K = max(viewportRows, 8) — the integration-notes formula, acknowledged as convention. We file a new FB to expose `BandStartRow`/`BandLength` (graft from leverage; §3 item N1) so this becomes contractual; until then risk R5's M1 spike covers it. In-band scroll is a zero-raster composite slide; band re-anchor re-rasters once (~2.2 ms measured at 200×60). De-realized presenters get `TearDown()` (INPC-leak rule).

**Presenters** are render-only leaf-block elements, every one `IsRenderBoundary = true`, so a keystroke re-rasters exactly one small zone: `ParagraphPresenter` (headings, def-list parts), `CodeBlockPresenter` (per-line-group child boundaries when large), `CalloutPresenter`, `RulePresenter`, `FrontMatterPresenter` (fold affordance; fold toggles height → extent invalidation), `ImageChipPresenter`, `TablePresenter` (§2.5), and per-item list / per-line blockquote presenters carrying container context (indent, quote depth) — a 300-item task list re-rasters one item per keystroke. Drawing goes through `RenderContext.DrawFormattedText(FormattedText, in Rect, …)` (`Cursorial.UI/RenderContext.cs:195,207`) with programmatically built `TextRun` inlines (required regardless — `TextMarkup` lacks underline-shape/hyperlink tags); `StyleQuantizer` degrades styled/colored underlines and attributes per color tier automatically inside `FrameRenderer`, so §7.1 link gating and §18 tier fallback need zero app branching.

**Styling/caps:** presenters expose classes (`Classes` public — `UIElement.Styling.cs:18`; class selectors first-class — `Styling/Selector.cs`): `md-h1..h6`, `md-code`, `md-callout-warning`, `:alternate` on table rows, `:active-block` for the §4.3 well tint. App theme XAML defines `Md.*` tokens in `ThemeDictionaries` keyed by `ThemeVariantKey`, layered over `CursorialDefaultTheme.LoadTheme()`; `caps-*` root classes (stamped per the full §18.2 catalog — `Styling/StyleEngine.cs:1449-1523`) drive declarative degradation exactly as §18 demands. Icons (checkboxes, callout glyphs, image chips) route through the framework `Icon` element (§18.4 ladder; FB-15's fourth tier tracked framework-side).

**Focus, caret, input:** `EditorControl` is the sole focusable element; presenters are inert visuals — no per-block focus churn. The caret is the real terminal cursor: `ITerminalCaretService.Publish(owner, col, row, CursorShape.BlinkingBar)` (`Cursorial.UI/TerminalCaretService.cs:14-24`; `Cursorial.Core/Output/CursorShape.cs:26`), `Clear`ed on focus loss or scroll-out — native blink, zero re-raster. Keys route to `EditController`, which consults the caret's block for context (table → `TableEditingController`); the keymap branches on `ProtocolCapabilities.KittyKeyboardProtocol` with legacy-safe defaults: dual `Ctrl+Space` gestures (FB-14.3), alternates for `Ctrl+Shift` chords and `Ctrl+/` (FB-14.1). Mouse hits resolve panel → block → run map → source position. Selection is a document-level source range (or the table cell-rect variant); presenters intersect it at draw time, so a selection change invalidates only presenters whose intersection changed.

`TextBox` is reused where it genuinely fits — link/image dialogs, find/replace bar, Options, the TaskDialog/file-dialog suite (FB-12) — never as the document surface: its buffer and undo are control-private, which would fork the model and break §4.2's single undo stack.

**Threading:** parse windows, autosave journaling (§12 snapshot = copying the line array — immutable strings make this cheap), and large-document find run off-thread, post back via `UIDispatcher.InvokeAsync`, and are epoch-validated on arrival.

### 2.4 WYSIWYG mapping

Layout of a block yields one run map per visual row: `Run(int SrcStart, int SrcLen /* block-relative */, int Col, int Width /* cells */, RunKind Kind)` with `Kind ∈ { Text, HiddenMark, RevealedMark, Synthetic }`. Hidden marks are zero-width runs at true source positions, so source↔rendered mapping is total (§4.2 [EDGE]). Synthetic runs either carry `SrcLen = 0` (table borders, quote bars, `↵`, checkbox icons — no caret stop) or **map to their marker source** (graft from leverage: a bullet run maps to `- `, so caret-left from item text lands before the item as exactly one stop). An image chip is a single run → one caret stop (§3.1 [EDGE]).

**Caret:** a source anchor; rendering position = block → visual row → run → cell (O(log rows + runs)). Horizontal motion walks grapheme-cluster boundaries within `Text`/`RevealedMark` runs, structurally skipping zero-width hidden marks; word motion segments the concatenated visible text (via `TextNavigation` once FB-1 lands; a small classifier over `GraphemeEnumerator` meanwhile). Goal column is stored in cells and re-applied per landing row; landing in a table snaps to the nearest cell (§3.1 [EDGE]). **Selection** maps a source range through run maps to cell spans per presenter — copy emits exact source for free (§3.4); partial-span cut/delete edits true offsets with spans re-derived on reparse (§3.2 [EDGE]).

**Reveal-on-edit (§4.1):** `ActiveBlockId` derives from the caret; the active presenter re-layouts with the active line's marks as `RevealedMark` runs (block-granular per the [DECISION]s: fences reveal, blockquotes reveal only the active line's `>`; emphasis spanning in reveals only the active line's portion — a per-row layout flag). **The revealed line never re-wraps.** It renders at natural width, horizontally slid within the block's box so the caret stays visible. **Grapheme-snapped clipping, designed explicitly** (resolving the fidelity judge's shared under-design objection): the visible slice is built by grapheme enumeration from the slide offset — slide offsets snap to cluster boundaries, and when a 2-cell cluster straddles a clip edge it is replaced by blank padding cells rather than half-rendered, preserving the whole-cell discipline (§1). Line count is invariant under reveal; no sibling shifts, no extent churn; a caret crossing blocks re-rasters exactly two zones. The M2 spike (R1) asserts all of this in cells under `UITestHost`.

**Find (§9)** searches source lines (the truth), maps hits through run maps for highlighting, and force-reveals a hit inside hidden marks (§9 [EDGE]).

### 2.5 Tables (§5)

`TablePresenter` owns a `TableModel`: per-row source line + pipe-aware, escape-aware `CellSpan[]`; per-column GFM alignment; and a **per-column width cache `(WidthCells, MaxContentWidth, CountAtMax)`** — all widths measured in cells via `GraphemeWidth` (§5.1 [CRITICAL]; never character counts). **Column widths are viewport-aware (revised 2026-07-05):** the model supplies each column's *natural* width (`NaturalWidth` = `MaxContentWidth` floored at `MinWidth = 3`, no cap) and the presenter resolves the final widths against the available viewport at measure time (`ResolveColumnWidths` — natural-fit vs. overflow-shrink, browser auto-layout), so a table stops wrapping just because it *could* — the fix for "empty viewport, still wrapping". The old `WidthCells` clamp `3..40` remains only as a viewport-unaware fallback. `CountAtMax` (graft, engine/leverage convergent) makes shrink detection O(1) and column re-measure O(one column) only when the unique widest cell was edited — the §5.5 acceptance ("deleting the widest content shrinks it back") with stated costs; a **resize** recomputes all columns and re-lays-out the table (a new trigger layered over WP5's content-edit reflow). Cells **word-wrap** at word boundaries (reusing `CaretNavigator.Wrap`/`WordWrap`, char-fallback only for an over-long word; `LayoutRow` stays the single owner of the wrapped-cell → visual-row mapping, risk a).

Editing routes through the standard pipeline: `TableEditingController` interprets Tab/Shift+Tab (last-cell Tab appends a row), Enter-commits-downward, Alt+↑/↓ row ops (legacy-wire-safe), and translates them into ordinary buffer edits on cell source ranges (pipes escaped, paste newlines→spaces per §3.4) or whole-line structural splices — each one sealed undo group. Intra-cell edits re-adopt the table block (Decision 4), and cell focus is (row, col) indices, surviving reparse.

**Raster economics:** per-row child presenters with `IsRenderBoundary = true` are a **committed M3 deliverable** (graft; both other proposals' judges demanded it), so an intra-cell keystroke re-rasters one row strip plus, on width change, the border band — not the table, never the document. Width changes recompute one column; height changes (row add, wrap change) shift prefix sums and fire `InvalidateScrollExtent` without repainting neighbors.

**Overflow (§5.6) — two orthogonal mechanisms (M3.WP6 delivered).** *(1) Cell mode* — a settable `OverflowMode` on `TablePresenter`/`MarkdownViewBridge` (default `Wrap`, mirroring `EditWrapEnabled`; the user command is M5). `Wrap` grows the row (multiple visual rows per logical row); `Truncate` keeps **one** visual row per logical row, clipping each over-wide cell to its column with a trailing `…` (U+2026, width 1) — a new `LayoutRow` branch (`LayoutRow` stays the single owner of cell→visual-row fragments, risk a), Wrap unchanged. The **focused** cell reveals its full content — the row presenter re-draws it un-truncated on top of the truncated base, overflowing rightward into its neighbours on that content row (spreadsheet-style) but clipped before the grid's right border; non-focused cells stay truncated. The Truncate caret map (`TableCaretMap`, overflow-parametrized) maps every cell at its **natural** full-content geometry — matching *both* the drawn prefix and the reveal — so click/caret land consistently; `ActiveSlide` stays 0 for a non-overflowing table. *(2) Column-window* — the automatic **edge fallback** (not a user mode) engaged in *either* cell mode when the grid can't fit the viewport even at `MinWidth` (`_metrics.Width > viewport`): the whole grid draws through a **presenter-internal horizontal offset** (the first-visible column's `BorderX`, in cells) applied in the row render as a translate-and-clip — column-granular, so a drawn column sits wholly inside the window. It **follows the caret** (`EnsureColumnVisible`, called from the bridge's reveal path scrolls the focused column on-screen), and the offset is surfaced through `ActiveSlide` so the caret publish subtracts it and a click adds it back — the `TableCaretMap` itself stays in unclipped grid cells, needing no rebuild when the window scrolls. **No nested `ScrollViewer`** (FB-6 sidestep, `Controls/ScrollViewer.cs:54-66,265`): `TablePresenter.MeasureOverride` reports the *windowed* width (≤ viewport) and `DocumentPanel.GetExtent()` pins the horizontal extent to the viewport, so the document's `ScrollViewer.Extent.Columns` == viewport and `HorizontalOffset` stays 0 — the document structurally never scrolls sideways (asserted).

**Cell-rect selection (§5.4, M3.WP8 delivered).** The selection stays an ordinary document **source range** (the model of truth); `DocumentCaret` *interprets* it as a whole-cell **rectangle** only when its anchor and active-caret offsets fall in **different cells of the same table** (`TableModel.CellRectOfRange` maps both offsets via `CellOfOffset`, normalizes to rows `[min..max]` × columns `[min..max]`; a same-cell selection returns null → the WP5 in-cell text highlight). The transition rule is a single guard in `DocumentCaret.CurrentCellRect`: both ends must be in the same block, or it is an ordinary document selection (the table portion then highlights per-cell over the covered span). The rect is surfaced through a new `ISelectionSource.GetCellRect(BlockId)` the way `GetSelection` already surfaces the range; `MarkdownViewBridge` wires it to `TablePresenter.CellRectProvider`, which forwards it to each `TableRowPresenter`. A row whose logical index is in the rect fills every selected cell's **full box** (`[BorderX(c)+1, BorderX(c+1))` — padding included) before drawing glyphs — `FillRectangle` with the `SelectionBrush` on the colour tier, Inverse spaces on NoColor — translated through the same column-window offset the grid draws with (so a windowed table highlights the on-window cells); interior dividers are **left as normal border glyphs**. **Copy** routes through the existing `SelectedText()` path: a cell-rect emits `TableEditingController.SubTableMarkdown` (top selected row → header, a delimiter synthesized from the *selected* columns' alignment, then the rest — valid GFM even when the rect excludes the model header). **Delete / Clear / Cut** route through `TableEditingController.ReplaceCellRect` — one contiguous structural splice from the top-left cell's content to the bottom-right's that blanks only each selected cell's `CellContentRange` and copies everything else (pipes, delimiter line, non-selected cells, terminators) verbatim — one sealed undo group, caret in the top-left cell.

**In-cell WYSIWYG inline formatting (Decision 9 per-cell reveal, retiring deferred-cleanup #4).** A cell's inline constructs render FORMATTED when the cell is inactive and RAW when the caret is in it, mirroring the prose machinery per cell. `TableModel.CellInlineRuns(row,col)` projects each cell's Markdig inline AST into cell-relative `InlineRun`s the same way `Block.RealizeInlineRuns` projects a block's — via a shared `InlineProjection` classifier (Markdig stays quarantined in the Document project). A `CellFormat` per cell then mirrors `RunMapBuilder.ClassifyMarks`/`ClassifyLine`: it classifies each content position as a hidden mark or visible content, accumulates the per-position inline style, and renders the marks-hidden **display** with its display↔source map (`IsPlain` short-circuits a cell with no inline construct, so plain tables pay only the cheap projection and render byte-identically). `TableModel.LayoutRow` gains an `activeColumn` param: the active cell (and every plain cell) wraps/truncates its RAW content as before, every other cell wraps/truncates its narrower `CellFormat.Display` — each formatted fragment carries `CellStyledRun`s (one per uniform-style, source-contiguous span), so `TableRowPresenter` draws the styled content (bold/italic/strike/code fill/link, the same styling `LeafBlockPresenter.StyleForContent` applies) and `TableCaretMap` emits one 1:1 stop per run (the hidden marks fall between runs and canonicalize onto their adjacent content). **Column widths are measured over the display width**, so hiding marks narrows an inactive cell and revealing the raw active cell reflows its row taller (only the active row's height varies; the grid width is stable). The reveal is wired through the same `MarkdownViewBridge.RevealActive`/`TablePresenter.EnsureColumnVisible`/`SetActiveCell` path that already tracks the active cell (both overflow modes now), which sets the active row's `TableRowPresenter.ActiveColumn` (re-deriving that one row) and drops the focus-dependent caret map.

### 2.6 Split view and raw mode (§4.2)

`EditorView` (control + panel + presenters) instantiates twice over the same `DocumentBuffer`, `BlockList`, `EditController`, and undo stack — "one document model, one undo stack" holds by construction; `BlockListChange` fans out to both panels. **Raw mode is a presenter mode:** `ViewMode.Raw` renders each block's source lines with token coloring derived from the same parse (classification is free), run maps become identity, and every caret/selection/find path is shared — mode-toggle caret preservation is the ordinary mapping with a trivial branch. Split layout is `Grid` + shipped `GridSplitter`; scroll sync maps the master pane's first visible source line through the peer's prefix sums, debounced one frame, feedback-guarded. Focus decides which pane publishes the terminal caret. Images stay placeholder-only in raw panes (§7.2 inert rule 3). Past ~10k lines the secondary pane re-renders on idle instead of per keystroke (§10.2's documented degradation).

## 5. Rejected alternatives (do not relitigate)

- **Bespoke incremental CommonMark/GFM parser (engine).** Rejected unanimously: it concentrates the entire §2 correctness surface (emphasis delimiter runs, link precedence, list indentation, lazy continuation — "the bulk of M2 parser effort" by its own admission) on unwritten code in the same milestone as reveal-on-edit and the caps layer; its kill-switch converges to this design after burning M2; and its two strongest anti-Markdig claims failed verification — Markdig spans are UTF-16 char offsets (it parses .NET strings), and §13 explicitly sanctions windowed/per-block reparse. Engine's *good* parts (re-adoption rule, lazy inlines, interval set, epochs, oracle harness, column-window overflow, `CountAtMax`) are grafted above.
- **Reveal that re-wraps and shifts subsequent blocks (leverage §d).** Direct §4.1 [EDGE]/acceptance violation ("must not reflow or shift any other line"); all three judges flagged it. We use the line-count-invariant horizontal slide.
- **One UIElement per *top-level* block (leverage §c).** A 10k-line changelog that is one ListBlock becomes a single zone (nothing virtualizes within one element); leaf-granularity presenters are the committed shape.
- **Absolute document offsets in run maps (leverage §d).** Every keystroke staleness-invalidates all realized maps after the edit point; block-relative spans make unedited blocks' maps valid by construction.
- **Block identity by content hash (hybrid/leverage as proposed).** Cannot match the block being edited — replaced by engine's (kind, first-unmodified-line) re-adoption; the hash survives only as a cheap secondary check for *moved* unchanged blocks.
- **`IsLogicalScroll = true` with 1-cell steps (engine).** A no-op: the `false` path already steps one cell / one viewport (`IScrollContentHost.cs:65-69`, verified).
- **Nested horizontal `ScrollViewer` for wide tables (hybrid as proposed, `Hidden` visibility).** Superseded by presenter-internal column-window rendering — no FB-6 exposure, no nested scroll focus/key routing questions.
- **`TextBox` as the document surface; `VirtualizingStackPanel`+ItemsControl as the primary host; Markdig trivia-mode round-tripping; command palette; multi-cursor.** Per-control undo/buffer forks the model (§4.2); the VSP route survives only as R5's fallback seam; trivia round-trip is unnecessary when the buffer is the artifact; the last two are spec-deferred (§14).
