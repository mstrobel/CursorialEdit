▸ Tables grow and shrink with content

1 row
┌────────┬─────┐ │ Item │ Qty │ ├────────┼─────┤ │ Apples │ 6 │ └────────┴─────┘
+ row added → grid extends
┌────────┬─────┐ │ Item │ Qty │ ├────────┼─────┤ │ Apples │ 6 │ │ Pears │ 12 │ └────────┴─────┘
+ typing a longer value → column widens, caret in cell
┌─────────────┬─────┐ │ Item │ Qty │ ├─────────────┼─────┤ │ Apples │ 6 │ │ Pears │ 12 │ │ Dragonfruit| │ 3 │ └─────────────┴─────┘
The showcase. A markdown table renders as a real box-drawing grid, and its borders are computed from content: adding a row extends the grid downward (new ├─┼─┤ + row + relocated └─┴─┘), and typing a longer value widens that column so every border cell stays aligned. The author never touches a pipe — they edit cells and the grid reflows, exactly like a GUI table but in characters.

Why it matters. Left: what a plain text editor shows — pipes, dashes, asterisks, backticks. Right: the same document in this editor — the table is a grid, **bold** is bold, `code` is tinted. In a terminal the rendered form lives in the same character grid as the source, so WYSIWYG isn't a separate preview pane — it's the editing surface itself.
